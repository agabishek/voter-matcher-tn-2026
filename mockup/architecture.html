System Overview

A fully data-driven political alignment engine. Zero hardcoded political content — all parties, axes, archetypes, languages and scoring parameters live in versioned config files.

🎯 What it does

Matches voters to political parties based on ideological alignment across 9 policy axes. Produces party match %, archetype, confidence score, and bilingual explanation.

🔧 Core principle

The application is a generic engine. Adding a new party, axis, language or updating a manifesto requires only config file changes — no code changes.

🌐 Scale target

100K+ concurrent users. Client-side scoring by default. Server-side fallback API. Static config served via CDN. 99.9% uptime target.

Key Design Decisions

Client-first scoring

All scoring runs in the browser. Server API is fallback only. Enables offline use and reduces server load.

Config-driven everything

Parties, axes, archetypes, languages, thresholds — all in JSON. Zero political content in code.

Hash-verified config

Every config file embeds its own sha256. App refuses to start if hash mismatches — tamper-proof.

Stateless API

No session state on server. Horizontal scaling trivial. Sessions live in browser localStorage only.

Why these choices?

Next.js App Router

Static generation for all screens. API routes for server-side fallback. Edge runtime for low-latency scoring API. Single repo for frontend + backend.

Client-side scoring

Scoring engine runs entirely in the browser using the config bundle. No server round-trip for the happy path. Reduces latency to <200ms trivially.

PostgreSQL only for analytics

No user data is stored. Only aggregated, non-identifiable analytics. PostgreSQL chosen for its JSON support and aggregation queries.

Service Architecture

Six core engines, all stateless, all reading from the Config Layer. Each engine has a defined interface and can be tested independently.

1. Config Loader

Startup

Loads all 6 config files, verifies sha256 hash of each, throws on mismatch. Provides ConfigBundle to all engines via React context.

interface ConfigBundle {
  parties: PartyRegistry
  axes: AxisRegistry
  archetypes: ArchetypeRegistry
  languages: LanguageRegistry
  questions: QuestionBank
  params: ScoringParams
  version: string
}

2. Questionnaire Engine

Session

Creates sessions, cluster-randomizes questions, tracks answers/skips, handles back-navigation, emits warnings.

createSession(config, lang): Session
selectOption(session, qId, optId): Session
skipQuestion(session, qId): Session
getProgress(session): Progress
getWarnings(session): Warning[]

3. Scoring Engine

Core

Sums Party_Weights and Axis_Weights, normalizes to percentages, computes dynamic confidence thresholds based on party count N.

compute(session, config): ScoreResult
normalize(raw): Record<string,number>
// Confidence: High = 100/N+12pp
//             Medium = 100/N+2pp
//             Low = below medium

4. Ideology Mapping Engine

Validation

Validates question bank against registries at startup. Checks collinearity (r < 0.70) and discriminating power (variance > 1.0). Emits weights on option selection.

validate(questions, config): ValidationResult
getWeights(optionId): WeightEmission
checkCollinearity(axes): CollinearityReport
checkDiscriminatingPower(): PowerReport

5. Profiling Engine

Classification

Classifies user into archetype from Archetype_Registry. Detects contradictions using axis independence pairs from Axis_Registry. Never reads party scores.

classify(axisScores, config): ArchetypeResult
detectContradictions(session, config): Contradiction[]
// Archetype independence: reads only
// Archetype_Registry, never party scores

6. Explanation Engine

Output

Generates all user-facing text dynamically from registries. No hardcoded strings. Supports any language in Language_Registry. Injects demographic context and track-record notices.

generate(result, archetype, config, lang): Explanation
// Sources: Party_Registry descriptions
//          Axis_Registry civic labels
//          Archetype_Registry descriptions
// Zero hardcoded explanation text

Data Flows

Three key flows: normal session, manifesto update, and adding a new party.

Flow 1: Normal User Session

1. App loads → ConfigLoader verifies all 6 config hashes → ConfigBundle injected into React context
2. User starts → QuestionnaireEngine.createSession() → cluster-randomizes questions → pins config version to session
3. Each answer → session updated in React state → synced to localStorage → IdeologyMappingEngine emits weights
4. Session complete → ScoringEngine.compute() → raw scores → normalize() → confidence (dynamic N-party formula)
5. Profiling → ProfilingEngine.classify() reads Archetype_Registry → primary + secondary archetype
6. Consistency → ProfilingEngine.detectContradictions() reads axis independence pairs → adjusts confidence if needed
7. Explanation → ExplanationEngine.generate() reads all registries → bilingual explanation assembled
8. Result rendered → Result_Dashboard reads Party_Registry for party count → renders N party cards dynamically
9. Share → POST /api/share → HMAC-SHA256 signed URL → monochrome share card generated
10. Session end → localStorage cleared → analytics incremented (aggregated only)

Flow 2: Manifesto Update

1. Operator runs → scripts/update-manifesto.ts --party DMK --file dmk-2026-v2.json
2. Script parses → new manifesto → recomputes Party_Weights for all DMK options in questions.json
3. Validation → IdeologyMappingEngine.validate() → collinearity check → discriminating power check
4. Hash update → new sha256 computed → embedded in questions.json and party-registry.json
5. Audit log → config_audit_log entry written: {prev_version, new_version, prev_hash, new_hash, diff_json}
6. Deploy → new config files pushed to CDN → ConfigLoader picks up on next app startup
7. In-progress sessions → complete using version pinned at session creation → unaffected
8. New sessions → use updated weights automatically

Flow 3: Add New Party

1. Operator runs → scripts/add-party.ts --id NEWPARTY --name-en "New Party" --name-ta "புதிய கட்சி"
2. Script adds → party entry to party-registry.json with governanceStatus, weightBasis fields
3. Script scaffolds → Party_Weight fields for NEWPARTY in all 30 questions (set to 0 as placeholder)
4. Operator fills → Party_Weights based on manifesto analysis (domain expert review required)
5. Validation → scripts/validate-config.ts → all checks pass
6. Hashes updated → both files rehashed
7. Deploy → app automatically renders N+1 party cards on Result_Dashboard
8. Confidence thresholds → auto-recalculate: High = 100/(N+1)+12, Medium = 100/(N+1)+2
9. Zero code changes → entire flow is config-only

API Reference

Four API routes — all stateless, all serverless-compatible. Scoring runs client-side by default; server API is fallback only.

GET /api/config

Public

Returns the public-safe ConfigBundle subset. Used by client to verify it has the latest config version. Does NOT include question weights (security).

// Response
{
  "version": "1.0.0",
  "parties": [...],       // Party_Registry (no weights)
  "axes": [...],          // Axis_Registry
  "archetypes": [...],    // Archetype_Registry
  "languages": [...],     // Language_Registry
  "params": {
    "questionCount": 30,
    "estimatedMinutes": 3
    // No weight ranges exposed
  }
}

POST /api/score

Fallback

Server-side scoring fallback. Used when client-side scoring fails (e.g., corrupted localStorage). Validates configVersion matches server's active version.

// Request
{
  "answers": {
    "q001": "q001_a",
    "q002": "q002_c"
  },
  "configVersion": "1.0.0",
  "language": "ta"

// Response
{

  "confidenceScore": "High",
  "archetype": { "primary": "equity_builder" },
  "explanation": { "primary": "...", "secondary": "..." }

Security Architecture

Politically sensitive system at scale — security is layered across config integrity, API protection, share card anti-forgery, and privacy.

Config Integrity

ThreatMitigation
Tampered config file skews resultsSHA256 hash embedded in each file, verified at startup. Mismatch = startup failure.
Unauthorized config changeConfig files in version-controlled repo. All changes logged in config_audit_log with operator ID.
Stale config served from CDNConfig version pinned to each session. Server validates configVersion on /api/score requests.

API Protection

ThreatMitigation
Reverse-engineer optimal answer sequencesRate limiting on /api/score (100 req/min). Question weights not exposed in /api/config response.
Automated bulk scoringBot detection: minimum session duration check, honeypot fields in questionnaire form.
Bulk fake share card generationRate limiting on /api/share (10 req/min per IP).
Forged share card URLsHMAC-SHA256 signing of all share URLs. Verification endpoint rejects tampered payloads.

Privacy

RequirementImplementation
No PII storedAnonymous_Mode by default. No session IDs, no IP logging, no device fingerprinting.
TLS 1.3All API routes enforce TLS 1.3 minimum. Configured at CDN/load balancer level.
localStorage cleared on session endSessionProvider clears localStorage on result view, browser close (beforeunload), and navigation away.
Analytics non-identifiableOnly day-level aggregates stored. No combination of fields can re-identify a user.
DPDPA compliancePre-session disclosure when analytics enabled. Specific disclosure of what is collected and why.

Share Card Anti-Misinformation

RuleEnforcement
No party colorsShareCardService uses monochrome palette only. Enforced in generator, not CSS.
No party logosShareCardService renders text labels only. No image assets from Party_Registry used.
Disclaimer always visibleDisclaimer text sourced from scoring-params.json, rendered as non-removable element in PNG.
Verify URL always presentMethodology URL embedded in every share card image and URL.
URL tamper detectionHMAC-SHA256 signature. Any modification to payload invalidates signature.

Cost Architecture

Designed for near-zero infrastructure cost. Total estimated monthly spend: $0-$1. All scoring runs in the browser — the server barely exists.

Why it costs almost nothing

Client-side scoring

All 6 engines run in the browser. Server API is fallback only — called maybe 0.1% of sessions. Vercel free tier handles millions of invocations/month.

Static config

All 6 config JSON files (~50KB total) are static assets on Vercel CDN. Zero serverless invocations for config reads. Cached in browser after first load.

Client-side share cards

Share card PNG generated via Canvas API in browser. Server only signs a URL string via Web Crypto API. No image processing on server.
